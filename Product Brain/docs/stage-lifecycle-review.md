# 阶段结束 & 阶段确认页 Review

**Files:** `04-StageDone.html`, `05-StageConfirm.html`, `07-StageDone2.html`, `08-StageConfirm2.html`
**Date:** 2026-06-22
**Product context:** Experiment R&D Platform — 阶段生命周期（场长提交 → 博士审核）

---

## 总评

阶段结束（StageDone）和阶段确认（StageConfirm）是阶段状态机的两个连续节点：场长提交阶段结束 → 博士审核（通过/驳回）。四页的 UI 风格一致，处理组切换 Tab、渐变统计卡片、确认/返回按钮布局等符合设计系统。但作为需要联动的两个角色页面，它们之间没有数据传递机制，都是独立页面各自维护状态。

---

## 发现清单

### P0

**1. 阶段结束 → 阶段确认之间没有数据流转**

`04-StageDone` 场长点击"提交"后跳转到 `05-StageConfirm`，但 `05` 不知道 `04` 提交了哪些数据——两个页面完全独立。博士看到的审核数据与实际提交无关。

> **建议：** 通过 URL 参数或 localStorage 传递阶段提交数据（至少传递实验编号 + 阶段编号 + 提交时间）。原型阶段可 mock 从 URL 读取。

**2. 阶段确认的"通过/驳回"操作无反馈**

`05-StageConfirm` 的确认/返回按钮没有绑定实际的审核逻辑。点击后应有状态变化和跳转。

> **建议：** 通过 → `ElMessage.success` + 跳转至下一阶段；驳回 → `ElMessageBox.prompt` 输入驳回原因 + 跳转回日常巡栏页。

### P1

**3. 阶段结束的统计卡片数据硬编码**

阶段汇总指标（结束重量、结束头数、阶段天数、死亡率）和处理组对比卡片的数据均为硬编码。没有从日常巡栏记录汇总计算。

> **建议：** 通过 URL 参数传入实验编号，使用 computed 从共享数据源计算阶段统计。

**4. 四页使用相同样式复制粘贴，存在样式漂移风险**

`04/05/07/08` 四页的 `<style>` 几乎完全相同（约 50 行 CSS 重复 4 次）。如果修改设计规范，需要同时更新 4 个文件。

> **建议：** 提取公共样式到独立 CSS 文件（`common.css`）并引用，或使用构建工具合并。

**5. 阶段结束缺少"提交前预览"**

场长点击"确认阶段结束"后直接提交（跳转），没有提供提交前的数据预览/确认步骤。

> **建议：** 增加提交确认对话框，展示阶段关键数据摘要，场长确认无误后再提交。

### P2

**6. 07/08 与 04/05 仅有阶段编号和数据差异**

第二阶段的页面（07/08）与第一阶段页面（04/05）代码结构完全相同，仅阶段名称从 `01-DRB` 变为 `02-ZZB`，示例数据不同。这种复制方式不便于扩展到 3+ 阶段的实验。

> **建议：** 合并为通用页面，通过 URL 参数 `stage` 区分阶段编号，动态加载对应数据。这消除了 04↔07、05↔08 的代码重复。
