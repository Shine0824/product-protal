# 实验详情页 Review

**File:** `experiment-detail.html`
**Date:** 2026-06-22
**Product context:** Experiment R&D Platform — 实验工作台（含分组→巡栏→阶段结束→归档→实验结束全流程）

---

## 总评

`experiment-detail.html` 是一个"超级页面"——将实验分组、日常巡栏、阶段结束、阶段归档、实验结束五个步骤全部集成在一个页面中，通过顶部的流程节点切换内容区。这个设计思路与 Product Brain 的阶段状态机架构高度一致，流程节点的视觉表现（active/completed/locked）也符合设计系统规范。但页面同时承载了分组表格（19 列可编辑）、日常巡栏表单、阶段统计、归档确认和实验结束五种差异巨大的 UI，导致代码复杂度极高（1000+ 行），维护和演示都有风险。

---

## 发现清单

### P0 · 演示前必须修复

**1. 多处关键操作使用 alert() 占位**

分组编辑（`editRow`）、数据上传（`handleUpload`）、分组完成（`completeGroup`）、取消完成（`cancelComplete`）、保存巡栏记录（`saveDailyRecord`）全部使用浏览器原生 `alert()` 弹出。在演示中这会严重损害产品感知。

> **建议：** 使用 `ElMessage` / `ElMessageBox` 替代所有 `alert()` 调用。关键操作（完成分组、确认归档）使用 `ElMessageBox.confirm` 带确认对话框。

**2. 分组表格"完成分组"后禁用编辑，但无视觉反馈**

`completeGroup()` 将分组步骤加入 `completedSteps`，表格输入框变为 `disabled`。但禁用态与非禁用态视觉差异极小（仅靠 Element Plus 默认的灰色），用户可能困惑"为什么我不能编辑了"。

> **建议：** 完成分组后，在表格上方显示明显的锁定横幅（"✅ 分组已确认 · 点击'取消完成'可修改"），并更改表格背景色以区分可编辑/只读状态。

**3. 实验编号硬编码为 `SWINE-NUR-DY-0126`**

无法从概览页传入实验上下文。这与其他所有页面的设计不一致——概览页的跳转逻辑 (`goToDailyRecords`) 通过 URL 参数传递实验编号。

> **建议：** 从 URL 参数读取实验编号（`new URLSearchParams(window.location.search).get('exp')`），并根据编号加载对应的实验数据。

### P1 · 建议优化

**4. 日常巡栏表单字段过于简化**

与 `03-DailyRecords.html`（prototype）的详细表单相比，此页面的日常巡栏仅 9 个字段（日期/时段/人员/正常头数/异常头数/体重/采食/温度/湿度/备注），缺少死淘明细、药品疫苗、腹泻评分等核心记录维度。

> **建议：** 至少增加死淘记录（数量/原因/性别）和腹泻评分，与 Product Brain 中定义的技术员记录维度对齐。或在此页面仅做"快速记录"，完整记录跳转至 DailyRecords 页面。

**5. 阶段统计数据全部硬编码**

阶段结束模块的 8 个统计卡片（阶段天数 45、初始 120 头、期末 118 头...）和底部的分组统计表格均为静态 HTML 写死的数据，不从 `tableData` 或 `dailyForm` 计算得出。

> **建议：** 阶段统计数据应从实际填写的分组数据和巡栏记录动态计算。原型阶段至少使用 computed 属性模拟计算逻辑。

**6. 数据统计概览表格使用原生 `<table>` 而非 Element Plus**

页面底部的"分组统计"和"分性别统计"两个表格使用原生 HTML `<table border="1">`，与上方 Element Plus `<el-table>` 风格不一致。

> **建议：** 统一使用 `<el-table>` 组件，保持视觉一致性。

**7. 阶段归档模块只有展示，无实际操作**

归档模块展示"归档内容"和"归档状态"两张静态卡片（均显示 ✓ 通过），缺少数据完整性检查的实际逻辑和异常记录的审核入口。

> **建议：** 至少模拟"数据完整性检查"的过程（如显示检查进度动画），并为"异常记录审核"提供可点击的查看入口。

**8. 缺少步骤间的导航引导**

完成一个步骤（如分组完成）后，流程节点自动更新，但页面不会自动滚动到下一个步骤或给出明确的"下一步"引导。用户可能停留在当前内容区不知所措。

> **建议：** 步骤完成后展示 toast 提示 + 自动切换到下一个步骤的内容区，或在当前区域底部显示"下一步 → 日常巡栏"的引导按钮。

### P2 · 锦上添花

**9. 分组表格 19 列，横向滚动体验差**

分组信息录入表有 19 列，在 1440px 屏幕上需要大量横向滚动。部分列（如性别_花色备注）使用频率低但占用了 140px 宽度。

> **建议：** 将低频列折叠到"更多信息"弹窗中，或使用列显示/隐藏切换。核心列（处理/区组/栋/单元/栏位/品系/初始头数/初始重量）控制在 8-10 列。

**10. 阶段流程节点在小屏幕上换行混乱**

流程节点使用 `flex-wrap: wrap`，在窄屏幕上节点会换行，导致流程可读性下降。

> **建议：** 增加横向滚动容器（`overflow-x: auto`），保持流程节点在一行。或在小屏幕上使用垂直时间轴替代水平流程。

**11. "实验结束"模块缺少阶段对比分析**

实验结束时仅显示"🎉 实验已完成"和简单的统计数字，没有阶段间对比（如 DRB vs ZZB 的增长曲线对比），也没有导出报告的入口。

> **建议：** 实验结束页增加阶段对比摘要图表（至少展示各阶段关键指标对比），并提供"生成完整实验报告"和"导出数据"的显式入口。
